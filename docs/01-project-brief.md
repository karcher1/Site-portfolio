# Project brief — Business Analyst RPG Portfolio

## Fixed concept

Create a one-page personal portfolio website for a Business Analyst in a professional 32-bit RPG-inspired style.

The portfolio should be creative, memorable, and game-inspired, while still suitable for recruiters, hiring managers, product teams, and stakeholders.

## Core metaphor

Business analysis as an RPG journey:

- unclear business problem = fog of war;
- discovery = exploration;
- requirements = quest log;
- process maps = world map;
- tools = inventory;
- skills = character stats;
- case studies = completed quests;
- achievements = unlocked outcomes;
- contact = start a new quest.

## Target audience

Primary:

- recruiters;
- hiring managers;
- product owners;
- delivery managers;
- founders or stakeholders looking for a business analyst.

Secondary:

- other analysts;
- product/design/development teams.

## Desired impression

The visitor should think:

- this person is structured;
- this person can explain complex things clearly;
- this person has personality and creativity;
- this person can work with stakeholders and development teams;
- this person is not just decorative — they are useful in delivery.

## Page structure

### 1. Hero / Player Profile

Purpose: quickly communicate identity and positioning.

Should include:

- name placeholder;
- role: Business Analyst;
- short positioning statement;
- RPG-like profile metadata;
- main CTA: View completed quests;
- secondary CTA: Start a quest / Contact me.

### 2. About / Character Backstory

Purpose: explain the business-analysis approach.

Should describe:

- working with vague problems;
- clarifying goals and constraints;
- mapping processes;
- writing requirements and user stories;
- aligning stakeholders and delivery teams.

### 3. Skills / Core Stats

Purpose: show key competencies in an RPG-stat format.

Example skills:

- Requirements Analysis
- Stakeholder Management
- Process Modeling / BPMN
- User Stories & Acceptance Criteria
- Backlog Management
- SQL / Data Analysis
- Product Discovery
- Communication & Facilitation

### 4. Tools / Inventory

Purpose: show the working toolkit.

Example tools:

- Jira
- Confluence
- Miro
- Figma
- SQL
- Power BI
- Postman
- Draw.io / Lucidchart

### 5. Case Studies / Completed Quests

Purpose: show practical value.

Case-study card structure:

- Quest name
- Challenge
- My role
- Artifacts
- Result
- Tools
- Status: Completed

Use anonymized examples if real company data is unavailable or under NDA.

### 6. Achievements / Unlocked Achievements

Purpose: show outcomes and professional strengths.

Examples:

- Turned vague requests into structured backlog items
- Reduced manual work through process improvement
- Aligned business and technical stakeholders
- Documented AS-IS / TO-BE processes
- Improved reporting clarity for decision-makers

### 7. Contact / Start a Quest

Purpose: make the next action obvious.

Should include:

- email placeholder;
- LinkedIn placeholder;
- Telegram placeholder;
- CV button placeholder;
- short final message.

## Content language

Default content language: Russian.

The code should make it easy to later add an English version, but bilingual functionality is not required in the MVP.

## MVP definition

The first good version is:

- one responsive page;
- all seven sections implemented;
- content stored in one editable data file;
- generated assets connected where available;
- no broken layout on mobile;
- successful production build.
